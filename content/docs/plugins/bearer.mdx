## Overview

The Bearer plugin provides Bearer token authentication by validating JWT access tokens from `Authorization` headers. It extracts tokens from the `Authorization: Bearer <token>` header format and validates them using the JWT plugin service, supporting both required and optional authentication modes.

### Features

- **Authorization Header Validation** вҖ” Extracts and validates `Authorization: Bearer <token>` headers
- **JWT Integration** вҖ” Uses the JWT plugin service for token validation
- **Dual Authentication Modes** вҖ” Required authentication (returns 401) or optional authentication (silent)
- **Cooperative Authentication** вҖ” Works alongside other auth plugins without conflicts
- **Configurable Header** вҖ” Customize the header name (default: `Authorization`)
- **Stateless Validation** вҖ” No server-side session storage required

---

## Configuration

### Standalone Mode

```toml
[plugins.bearer]
enabled = true
header_name = "Authorization"
```

### Library Mode

```go
import (
	bearerplugin "github.com/Authula/authula/plugins/bearer"
)

bearerplugin.New(bearerplugin.BearerPluginConfig{
	Enabled:    true,
	HeaderName: "Authorization",
}),
```

---

## API Reference

The Bearer plugin does not expose any HTTP endpoints. It operates as a middleware plugin that validates incoming requests via hooks.

---

## Database Schema

The Bearer plugin does not create any database tables. Token validation is performed statelessly using the JWT plugin service.

---

## Plugin Capabilities

The Bearer plugin provides the following capabilities:

- **bearer.auth** вҖ” Required authentication hook that validates Bearer tokens and returns 401 if invalid or missing
- **bearer.auth.optional** вҖ” Optional authentication hook that validates Bearer tokens if present and silently continues if invalid or missing

### Flow Description

**1. Header Extraction** вҖ” The plugin extracts the `Authorization` header from the request and parses the `Bearer <token>` format.

**2. Token Validation** вҖ” The extracted token is validated using the JWT service, which verifies the signature, expiration, and blacklist status.

**3. Success Path** вҖ” Valid tokens result in the `UserID` being set in the request context, allowing the request to proceed.

**4. Failure Path** вҖ” Invalid or expired tokens return a `401 Unauthorized` response with an appropriate error message.

### How to Use

Enable Bearer authentication on routes by adding the appropriate hook ID to route metadata:

**Required Authentication (returns 401 if no valid token):**

```toml
[[route_mappings]]
path = "/api/protected"
method = "GET"
plugins = ["bearer.auth"]
```

**Optional Authentication (works whether authenticated or not):**

```toml
[[route_mappings]]
path = "/api/public"
method = "GET"
plugins = ["bearer.auth.optional"]
```

### Client Request Format

Clients must send JWT access tokens in the `Authorization` header using the Bearer scheme:

```http
Authorization: Bearer eyJhbGciOiJFZERTQSJ9...
```

### Cooperative Authentication

The Bearer plugin implements cooperative authentication вҖ” if a request already has a `UserID` set by another authentication plugin (e.g., Session), the Bearer plugin will skip its validation. This allows multiple auth methods to work together seamlessly.

---

## Dependencies

- **JWT Plugin** вҖ” Required: the Bearer plugin validates tokens using the JWT service
- **JWT Service** вҖ” Must be registered in the service registry by the JWT plugin

---

## Client Plugin

If you're using the [Authula SDK](https://github.com/Authula/authula-node-sdk), add the plugin to the SDK like so:

```typescript
import { createClient } from "authula";
import { BearerPlugin } from "authula/plugins";

export const authulaClient = createClient({
  url: "http://localhost:8080/auth",
  plugins: [
    // other plugins...
    new BearerPlugin({
      headerName: "Authorization", // optional, defaults to "Authorization"
    }),
  ],
});
```

## Security Features

- **Strict Format Validation** вҖ” Enforces proper `Bearer <token>` format
- **JWT Validation** вҖ” Leverages the JWT plugin's comprehensive token validation (signature, expiration, blacklist)
- **Clear Error Messages** вҖ” Returns descriptive 401 errors for debugging
- **No Token Storage** вҖ” Stateless validation вҖ” no server-side session storage required
- **HttpOnly Not Required** вҖ” Unlike cookie-based auth, Bearer tokens are sent explicitly in headers
